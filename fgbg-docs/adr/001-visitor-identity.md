# ADR-001: 访客身份替代账号系统

## 状态

**accepted**

## 上下文

mdocs 面向小团队或个人的轻量级知识库。传统账号系统（注册/登录/密码/邮箱验证）带来以下负担：

- 用户需要记忆密码或依赖第三方 OAuth。
- 服务端需要密码哈希、找回密码、会话管理。
- 对于「看一眼文档」的场景过重。

同时，项目作者希望保持「无账号」的轻量体验，类似 Etherpad 或某些协作白板。

## 决策

采用**访客身份（Visitor Identity）**机制：

1. 用户输入一个昵称即可使用。
2. 后端生成 `visitor_id`（UUID，公开标识）+ `visitor_token`（高熵随机字符串，凭证）。
3. 浏览器保存原始 token（`localStorage`），每次请求通过 `x-visitor-token` header 发送。
4. 后端只存 `SHA-256(token_hash)`，永不保存原始 token。
5. 提供访客合并工具（`pnpm mdocs visitor migrate`），处理浏览器缓存丢失后的身份恢复。

## 选项对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| 传统账号（密码） | 成熟、可跨设备 | 实现重、用户体验差、需要密码找回 |
| OAuth（GitHub/Google） | 用户无新密码 | 依赖第三方、需要配置、不适合局域网部署 |
| **访客 Token（选中）** | 零门槛、服务端轻量、局域网友好 | Token 丢失即身份丢失、需合并工具、XSS 可窃取 token |

## 后果

### 正面

- 新用户可在 3 秒内开始写作。
- 服务端无需密码学基础设施。
- 单进程部署即可运行，适合本地和小团队服务器。

### 负面

- **Token 丢失 = 身份丢失**：浏览器清除数据后，用户变成全新访客。
  - 缓解：提供访客迁移 CLI 工具，由管理员合并。
- **XSS 风险**：`localStorage` 中的 token 可被恶意脚本窃取。
  - 缓解：无第三方脚本加载（自建部署），输入内容使用 Lexical 编辑器（自带 sanitization）。
- **无法主动登出**：没有「退出登录」概念，只能清除浏览器数据。
  - 接受：符合轻量定位。
